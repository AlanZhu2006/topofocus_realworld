# Step 7: Hard Pruning + Room Tokens
# ===================================

## Context

Steps 1-6 are complete. Step 6 computes active_patches (~64-256 patch indices out of 1600)
deterministically using room mask (frontier room + neighbor rooms + delta patches). However,
these active patches are only logged/visualized -- never actually used to prune the VLM input.
Step 5 attention is broken (highlights corners, not frontiers), but Step 6 and Step 7 are
purely geometric and don't depend on it.

Goal: Before the LLM forward pass, replace 1600 vision tokens with [R room tokens] + [K active
patches] where R~3-8 and K~64-256. The ViT still processes the full image; pruning happens
between ViT output and LLM input.

## New File

### CogVLM2/basic_demo/vision_pruning.py

Core pruning module with:
- _pruning_state dict -- global state holding active_patch_indices, room_patch_groups,
  num_pruned_patches, index_mapping
- prune_vision_features(images_features, active_indices, room_groups) -- takes [B, 1600, D]
  ViT output, returns [B, R+K, D] (room mean-pools + gathered actives)
- set_pruning_params(api, rpg) / clear_pruning_params() -- called before/after model.generate()
- install_pruning(model) -- one-time setup at server startup:
  1. Registers a forward_hook on model.transformer.vision to prune the output tensor
  2. Monkey-patches model.transformer.forward to use R+K instead of hardcoded 1600 for num_patches
  3. Monkey-patches model.prepare_inputs_for_generation for same num_patches override

Why monkey-patch? modeling_chatglm.py hardcodes num_patches = (image_size // patch_size // 2)**2
= 1600 in two places. We can't set a config to produce arbitrary R+K. The vision hook changes
the tensor shape; the monkey-patches make the splicing logic use the actual shape.

## Key Design Decisions

1. Position encoding is trivial: GLM-4V gives all 1600 vision tokens the same position_id
   (repeated). After pruning, same pos_id repeated R+K times -- no per-patch position needed.

2. First step = no pruning: active_patches_state[j] = None -> client sends no pruning params
   -> server uses all 1600 tokens (baseline behavior).

3. Only the Decision VLM call is pruned: Caption, Perception, and FN calls use RGB images
   (not the map), so pruning doesn't apply.

4. Room tokens use current room_to_patches: Even though active_patch_indices come from the
   previous step, the room pooling groups use the current step's mapping (matches the image).

5. ViT always runs on full image: Pruning is between ViT output [B, 1600, D] and LLM input
   splice -- no ViT modification needed.
