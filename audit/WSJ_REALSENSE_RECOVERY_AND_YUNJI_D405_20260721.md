# wsj RealSense recovery, Yunji D405 camera integration, and real board-based cross-robot calibration — 2026-07-21

## wsj: RealSense camera unusable after reboot — root-caused and fixed (not hardware)

After a physical reboot, wsj's RealSense D435i enumerated over USB but failed to stream: `control_transfer returned error ... Resource temporarily unavailable, number: 11` followed by `Depth/IR stream start failure, Hardware Error`. Initial hypotheses (USB hub routing, a bad port, a physically damaged camera) were tested and eliminated one at a time:
- Reseated the cable on the original Genesys USB3.2 hub port: same failure.
- Tried the Jetson's OTG micro-USB port: kernel logs (`usb-conn-gpio ...usb2-0:connector: repeated role: host` + `VDD_5V0_SYS: Underflow of regulator enable count`) showed the connector itself electrically bouncing — and this port is USB2.0-only (480Mbps) regardless, inadequate for the camera's USB3 multi-stream profile.
- Tried another physical port: failed to enumerate at all.
- Back on the original hub port: camera enumerated fully (5 video interfaces + HID) but streaming still failed identically.
- Isolated single-stream tests: color-only recovered after one transient error and produced stable real data (20-28Hz); depth-only (minimal load) still failed identically — this ruled out "combined bandwidth too high" as the sole cause, since even a single small stream failed.

User pushed back correctly ("I only unplugged/replugged, this shouldn't be hardware damage"), prompting reconsideration. Root cause found, both purely software/config, neither requiring any hardware fix:
1. `/sys/module/usbcore/parameters/usbfs_memory_mb` had reset to the kernel default of 16MB after reboot — a well-known, too-small value for RealSense's USB3 isochronous multi-stream transfers. Raised to 1000.
2. USB autosuspend (`power/control`) was `auto` for both the camera and the hub carrying it — raced with active streaming. Set to `on` for both.

After both fixes, isolated streams were reliable; a subsequent full clean run of the complete 6-stream tinynav profile (depth+color+dual-infra+gyro+accel, sync+align) ran stably for 100+ seconds with real, continuous frame delivery to the hub (30 frames sent, all `ack=accepted`, expected 2Hz cadence) — the earlier full-profile failures were most likely the fixes not yet being (re)applied after intervening reboots, not a real bandwidth ceiling.

**Persisted so this survives future reboots** (previously it did not — `usbfs_memory_mb` and `power/control` both silently reset every boot):
- `systemd` oneshot service `usbfs-memory-fix.service`, enabled, sets `usbfs_memory_mb=1000` at boot.
- `udev` rule `99-realsense-usb-power.rules`, matching by vendor/product ID (camera `8086:0b3a`, Genesys hub `05e3:0625`) so it re-applies automatically regardless of which physical port the device ends up on — sets `power/control=on` on device add.

Also found and fixed during this recovery: the reverse SSH tunnel routing wsj's sender to the hub (port 18089→8088) died silently after every wsj reboot/network drop (process alive, forwarding dead) and needed manual recreation each time; and `nohup ... & disown` was unreliable for launching the long-running rehearsal wrapper script over wsj's flaky network tonight (the whole script would die with no error). Switched to launching the wrapper itself inside a persistent `tmux` session, which survives SSH disconnects.

## Yunji: new Intel RealSense D405 integrated as `--camera-source local-realsense`

User mounted a second, physically different camera (D405, connected directly to nyush-nuc over USB, not the chassis) for a higher vantage point than the existing low-mounted `camera_front_up`. New capability added to `yunji_sender.py`:

- `LocalRealsenseCamera`: opens one `pyrealsense2` pipeline, `rs.align(rs.stream.color)` for depth-to-color alignment using the device's own real factory calibration (not the hand-rolled `reproject_rgb_onto_depth_grid`/hardcoded `K_DEPTH`/`K_RGB` used for the chassis camera). A background thread owns the single `wait_for_frames()` call (there can only be one reader of a physical camera) and continuously captures at the camera's native rate; the main sender loop's `read()` is a thread-safe getter of the latest frame, decoupling the 2Hz upload cadence from the camera's real frame rate. The same background thread independently pushes a dashboard preview at its own faster, throttled rate (`--preview-max-rate-hz`, default 10Hz) — this is a different shape from the earlier-reverted `RgbStreamThread` experiment for the chassis camera (that one added a second rosbridge connection that contended with the main loop's own fetch of the same flaky server; here there is exactly one reader of the camera).
- New `--camera-source {rosbridge,local-realsense}` switch; the chassis rosbridge path is unchanged and still the default.
- Environment setup required on nyush-nuc from scratch: `pip install pyrealsense2`; two separate USB permission fixes were needed before the device was usable by a non-root process — a udev rule for the raw USB bus device node (matching vendor `8086`, `MODE=0666 GROUP=plugdev`), and separately adding the user to the `video` group (librealsense actually reads the `/dev/video*` V4L2 nodes uvcvideo creates, which are a different permission domain from the raw USB device node).

**Camera-to-base_link extrinsic corrected for the new mount.** The existing `MEASURED_T_BASE_LINK_CAMERA` was real-`/tf`-measured for the old chassis camera specifically; it does not apply to the D405, which is bolted to nyush-nuc, off the chassis's own published frame tree (no `/tf` edge exists to measure it against). Added `MEASURED_T_BASE_LINK_CAMERA_D405`, built from a user-reported verbal measurement (12.7cm right / 20.3cm forward / 33.5cm up from base_link's origin, mounted level with no pitch/yaw/roll) converted into base_link's REP-103 axes and composed with the fixed body-frame-to-optical-frame rotation implied by "mounted level" — **not independently verified against a real measurement device**, only against the user's own description; `pose_to_matrix()` now takes the extrinsic as a parameter, selected by `--camera-source` at the call site, so the chassis camera's own real `/tf`-derived extrinsic is untouched.

**D405 depth quality at room-scale distance, measured for real (with a self-caught methodology bug).** First measurement pass showed alarmingly high noise even within the D405's rated close-range sweet spot, which turned out to be a bug in the analysis: zero (no-reading) frames were being averaged in as if they were valid zero-depth readings, biasing both mean and stddev. Corrected (per-pixel mean/std computed only over frames where that pixel had a real reading, requiring at least half the frames valid): real temporal noise is ~5-11mm from 0-1.5m, growing to ~58-72mm at 1.5-3m, and effectively unusable (~790mm median) beyond 3m — physically consistent with a genuine stereo camera and the D405's notably short 18mm baseline (vs. the D435i's 50mm), not evidence the sensor is broken. Practical implication: fine for typical indoor room/corridor scale (most content <2m), poor for open/long spaces.

## New tool: real cross-camera extrinsic calibration via a shared visual target

`calibrate_shared_frame.py` (existing) only supports two cases: robots physically coincident, or a manually pre-measured `--offset-file`. Neither fit today's actual setup — both robots placed at *different* positions around one shared calibration board (7x10 symmetric circle grid, 40mm spacing, 24mm circle diameter). New tool added: `hub/tools/calibrate_camera_offset_via_board.py` — detects the circle grid in each robot's own RGB frame (`cv2.findCirclesGrid`, upscaling the image 4x before detection, since the default blob detector was unreliable at the circles' native pixel size in both robots' photos), solves PnP per camera using each camera's own real intrinsics/distortion, and composes the two board-relative poses into the one `T_referenceCamera_otherCamera` offset `calibrate_shared_frame.py`'s existing `--offset-file` already expects — no changes needed to that script.

Caveat stated plainly in the tool's own docstring: both PnP solves use each camera's COLOR optical frame; this is only exactly correct if each robot's published pose also represents its color optical frame (confirmed true for the new Yunji D405 path; not independently re-verified for wsj's TinyNav sender here).

Run for real against a synchronized photo pair (wsj: board detected at 2.447m; Yunji: board detected at 1.174m; both used their real, non-fabricated camera intrinsics — wsj's from its live `camera_info` topic, Yunji's from its live `pyrealsense2` profile). Applied to `calibrate_shared_frame.py` using wsj's currently-live pose stream (the IMU-fix sender, see below) as reference. **Verified independently, not just internally consistent**: the two robots' own independently-built map origins (each daemon separately picks its own bounding box from its first observation) ended up only 1.415m apart — matching the 1.423m camera-to-camera distance the board calibration computed, a real quantitative confirmation the calibration is placing both robots' maps in a physically sensible shared frame, not the coincident-assumption approximation used previously.

Corresponding daemons restarted cleanly with `--start-after-sequence` (skip pre-calibration history) and the existing `--expected-transform-version` guard (rejects any observation whose transform_version doesn't match, preventing silent old/new frame mixing).

## Stationary drift check: real asymmetry found, one likely confound identified

Recorded a pose baseline for both robots, waited 6 minutes (both robots reported stationary), and diffed. Yunji: ~0.2mm drift — essentially zero, as expected. wsj: ~22.7cm drift, concentrated in Z (-20.8cm) and Y (-9cm), with `localization_state=DEGRADED` throughout. User's explanation after the fact: the Go2 physically lay down during the window (a real posture change, not a rigid-body no-op), which would plausibly explain a height change of this rough magnitude. **This explanation was not independently re-verified** (no before/after posture photo or log was captured) — user chose not to redo the test. Recorded here as: a real measured number (22.7cm/6min), with a plausible but unconfirmed explanation, not a confirmed non-issue. If wsj's static drift is retested later, the robot's posture must be confirmed fixed (e.g. standing-locked) for the whole window for the result to cleanly isolate SLAM/IMU drift from posture changes.
